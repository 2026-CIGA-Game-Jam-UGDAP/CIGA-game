四、Debug 时怎么给我信息

效果不对的时候，按这个顺序给信息，一次解决：

🐛 Debug 信息模板

① Console 报错 ← 最重要！有就给，没有说"无报错" [粘贴完整报错文本 / 截图] ② 现象对比 预期：[你预期的效果描述] 实际：[实际发生的效果描述] ③ 刚改了啥 ← 列举最近改过的相关东西 - 新增了 xxx 脚本 - 把 xx 的 yyy 字段改成了 zzz ④ 你在 Inspector 看到的具体引用 ← 快速确认引用链 "Player 上的 CameraShake 组件：impulseSource = xxx（已拖/未拖）"
⚠ 不要转述 Console 报错！ "它说有个空引用" 和实际的 "NullReferenceException: CameraShake._impulseSource" 是完全不同的信息量。
给我报错文本，我不需要你翻译。

五、grill-me 的决策框架 — 我盘问你的逻辑

当我说"这个需要你决策"时，我的盘问逻辑是按这条树走的。 你提前知道，就能抢在我问之前回答，省一轮对话。

5.1 技术决策树（优先级从高到低）

① 网络策略
   └─ 这个效果需要同步给其他玩家吗？
        ├─ 要 → 要考虑 [Command]/[ClientRpc]/SyncVar
        └─ 不要 → Client-only，跳过网络层

② 放哪
   └─ 这个组件挂在哪个 GameObject 上？
        ├─ Player 身上 → 注意 isLocalPlayer 限制
        ├─ Camera 上 → 要考虑场景切换后会不会丢失
        ├─ Canvas 下 → 是 Overlay 还是 Camera 模式？
        └─ 场景中 → 生成/销毁生命周期谁管理

③ 触发方式
   └─ 谁调用这个方法？
        ├─ PlayerController 自己 → 内部直接调
        ├─ 其他脚本（Health/Weapon）→ 要暴露 public 方法
        └─ 事件驱动 → 需要注册回调或 GetComponent

④ 可调参数
   └─ 哪些值需要暴露在 Inspector 里？
        ├─ 调参调手感 → [SerializeField] 拖到 Inspector
        └─ 数值不变 → 硬编码，减少 Inspector 噪音

⑤ 性能
   └─ 这个效果高频调用吗？
        ├─ 每帧执行（如 ItemHover）→ 避免 Instantiate/GC
        └─ 偶发触发（如 DeathBurst）→ 怎么简单怎么写

5.2 设计决策树（grill-me 会盘问你的）

① 这个效果的核心目的是什么？
   ├─ 操作反馈（"我刚刚做了个动作"）→ 快速、干脆、不遮挡
   ├─ 信息传达（"我被打了，从哪边"）→ 精准、清晰、可读
   ├─ 情绪烘托（"这一击很重"）→ 夸张、慢动作、暂停感
   └─ 社交信号（"我杀人了！"）→ 高调、全屏、所有人看到

② 和已有系统怎么衔接？
   ├─ 替换现有效果 → 加开关，旧效果先保留
   └─ 叠加在现有效果上 → 确认不冲突

③ 你觉得"够好"的标准是什么？
   └─ 不要追求完美——Jam 的"发布标准"比工程低很多

六、最优合作节奏

一个 burst 45 分钟，产出 3-5 个效果：

你（5 分钟）  →  把 3-5 个效果按四件套写给我
我（10 分钟） →  一次性写完所有脚本
你（20 分钟） →  切 Unity，一次拖完所有引用 + 粒子参数
你（5 分钟）  →  跑一次，确认哪些工作、哪些不工作
我（5 分钟）  →  看到报错，一次修完
不要"写一个配一个"。
攒够再配的连续操作手感，比断断续续切 Unity 快 2 倍。
你的瓶颈是拖引用，不是我等脚本。

七、这文档怎么用

  1. CLAUDE.md 放稳定信息（引擎版本、游戏类型、代码风格偏好）—— 你维护，几乎不动

  4. 决策分歧 时，按第五章的决策树走，你选分支我执行
  5. 磨合出的新偏好 告诉我，我记到 Memory 里，下次自动用
