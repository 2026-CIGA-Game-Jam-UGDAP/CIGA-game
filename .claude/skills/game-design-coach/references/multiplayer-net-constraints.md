# 🌐 Mirror 网络约束参考

> 读取此文件的场景：需要理解 Mirror 的技术限制来指导设计决策时。
> 目标读者：策划。不需要懂 C# 细节，但需要理解哪些设计会给网络层带来麻烦。

---

## 一、核心概念：策划需要知道的

### 什么是 Mirror？
Mirror 是 Unity 的一个网络库，采用 **服务器权威（Server Authority）** 模式：

```
客户端 A ──[Command]──→ 服务器 ──[ClientRpc]──→ 客户端 B
   ↑                          │                       ↑
   │                          ▼                       │
   └──────────[ClientRpc]─────┘───────────────────────┘
   
- Command：客户端告诉服务器"我要做某事"
- ClientRpc：服务器告诉所有客户端"发生了某事"
```

### 这对策划意味着什么？

**❌ 不要设计需要"客户端自己算"的机制：**
```
❌ "每过 5 秒恢复 1 点生命"（客户端计时 → 不同步）
✅ "回合结束时恢复 1 点生命"（服务器在回合结束时统一算）

❌ "点击敌人时显示攻击范围和伤害预览"（客户端要算路径和伤害 → 写两份逻辑）
✅ "选择目标后显示有效目标"（服务器告诉客户端哪些目标有效）
```

**✅ 好的设计是"事件驱动"的：**
```
好的设计：玩家 → 出牌 → 服务器收到 → 服务器结算 → 告诉所有人
不好的设计：玩家 → 出牌 → 客户端放动画 → 客户端算伤害 → 告诉服务器
```

### 48h 黄金法则：服务器做所有重要的事情

```
服务器负责：                 客户端负责：
✓ 发牌                       ✓ 显示手牌
✓ 计算伤害                   ✓ 播放动画
✓ 判定谁赢了                 ✓ 显示UI
✓ 维护回合顺序               ✓ 播放音效
✓ 管理牌堆洗牌               ✓ 输入处理
```

> 如果某个逻辑是"重要的"（影响游戏结果），它必须在服务器上运行。

## 二、卡牌游戏的网络模型

### 推荐架构：Room-Based（房间制）

```
大厅 ─→ 房间1（3-4人） ─→ 游戏
     ─→ 房间2（3-4人） ─→ 游戏
     ─→ 房间3（3-4人） ─→ 游戏

Mirror 提供：NetworkRoomManager + NetworkRoomPlayer
```

### 房间模式下的同步流程

```
1. 玩家加入房间 → NetworkRoomPlayer 自动创建
2. 房主点击开始 → 所有人心跳同步跳转游戏场景
3. 游戏中：
   玩家 A 出牌 → [Command] SendPlayCard(cardId, targetId)
               → 服务器验证（A 的手牌有这张牌吗？目标有效吗？当前是 A 的回合吗？）
               → 服务器执行（扣血、摸牌、弃牌）
               → [ClientRpc] 通知所有人状态变化
4. 游戏结束 → 服务器判定胜利 → 回到房间
```

### 策划需要知道的同步基础知识

**SyncVar（自动同步变量）**
```
适合：血量、回合数、当前玩家编号、游戏阶段
在服务器上修改 → Mirror 自动同步到所有客户端
不需要写额外代码
```

**SyncList（自动同步列表）**
```
适合：手牌、牌堆（作为 Card ID 列表）、弃牌堆
在服务器上增删 → 客户端自动收到变更通知
```

**Command（客户端→服务器调用）**
```
适合：玩家的所有操作（出牌、结束回合、使用技能）
客户端调用 → 服务器执行
```

**ClientRpc（服务器→所有客户端调用）**
```
适合：通知所有人发生了某事（某玩家出牌了、某玩家受伤了）
服务器调用 → 所有客户端执行
```

### 这个架构下 DO 和 DON'T

```
✅ DO：
- 用 Command 发送"意图"（"我要出牌"）
- 用 SyncVar 同步"状态"（"血量=3"）
- 用 ClientRpc 广播"事件"（"A 出了飞刀"）

❌ DON'T：
- 不要在客户端生成随机数（"猜拳"这种需要服务器随机）
  → 服务器洗牌，客户端只读
- 不要在客户端做关键判定（"这张牌打中了没有"）
  → 服务器算命中
- 不要依赖客户端的输入时间（"3 秒内出牌不然 pass"）
  → 服务器计时，服务器判定超时
```

## 三、时间有关的策划注意

### 回合计时器

如果要做回合计时（"30 秒不出牌自动 pass"）：
```
架构选择：
- 方案 A（简单）：服务器设计时器，超时自动处理
  → 同步问题少，策划和程序都好理解
  → 缺点：计时结束时的操作要处理边界情况

- 方案 B（困难）：客户端显示倒计时，到了自动发送 pass
  → 客户端之间的倒计时可能差几秒
  → 不要用，48h 不值得

推荐：方案 A
```

### 同时行动（"所有人一起出牌"）

如果设计所有玩家同时出牌：
```
技术要点：
1. 每个客户端提交一个 Command（包含出牌选择）
2. 服务器收集所有玩家的 Command
3. 等所有玩家都提交 OR 超时
4. 服务器统一结算

策划注意：
- 需要约定一个"收集截止"的时间点
- 需要设计"没出牌"的默认行为（自动 PASS？自动出最左边的牌？）
- 结算顺序需要明确（按玩家ID顺序？按某个速度属性？）
```

## 四、网络延迟问题

### 卡牌游戏的网络特点（好消息！）

卡牌游戏是 **回合制** 的 → 对延迟 **极不敏感**。

```
FPS 游戏需要：< 50ms 延迟
卡牌游戏只需要：< 1000ms 延迟（1秒）

原因：你出牌 → 服务器处理 → 其他客户端收到
整个流程 1 秒对卡牌体验完全没影响
```

### 唯一要注意的：连接断开

```
比延迟更重要的是"连接断开"的处理：

- 简单方案（推荐 48h）：玩家断开 → 该玩家掉线 → 游戏继续（少一个人）
  → 技术上最省事，但剩下玩家体验受影响

- 进阶方案：玩家断开 → 等待 30 秒 → 掉线
  → 需要做"重连"逻辑，48h 不建议

推荐：简单方案，告诉玩家"不推荐断连"
```

## 五、FQA — 策划常问的技术问题

### Q: 可不可以做一个"所有人同时抢牌"的机制？

**A: 可以做，但两种设计方案：**
1. 所有人同时提交 → 服务器按顺序处理 → 如果牌被抢走了后面的人拿不到
   → 最简单，玩家提交时不知道结果（有点赌博感）
2. 回合交替做 → 当前玩家出牌，其他人只能响应
   → 最可控，传统卡牌的做法

**推荐 48h：方案 2。** 方案 1 需要处理"显示延迟"的问题（A 先抢到了但 B 的屏幕上还在等结果）。

---

### Q: 可以做一个"偷看对方手牌"的技能吗？

**A: 技术上可以做，但 48h 不值得：**
- 需要实现"只发 A 不看 B"的网络同步（Server→A 单独的消息）
- 需要 UI 上的特殊处理（偷看弹窗）
- 对游戏平衡影响大

**替代方案（48h friendly）：**
- "随机弃对手一张牌" — 不需要看，直接弃
- "猜对手手牌" — "猜测对手手牌中是否有 X 类型牌"，猜对有效

---

### Q: 可以支持观战模式吗？

**A: 不做。** 观战需要 Observer 角色，Mirror 的 Room 模式对这个支持不好。48h 不值得。

---

### Q: 最多支持多少人对战？

**A: 4 人。**

对于 48h jam 的卡牌游戏：
- 2 人 → 测试最方便（甚至单机双开测）
- 3 人 → 可选
- **4 人 → 推荐的上限**
- 4 人以上 → 网络同步复杂度指数上升 + 等待时间变长 + UI 放不下

> 策划注意：设计时以 3-4 人为基准，但保证游戏在 2 人时也可玩（可以支持测试）。

## 六、48h 的技术红线（策划必须知道）

| 技术特性 | 时间成本 | 建议 |
|---------|---------|------|
| 基础出牌 + 伤害 + 死亡 | ~3h | ✅ 必须做 |
| 手牌同步（发牌/摸牌/弃牌） | ~2h | ✅ 必须做 |
| 回合管理 | ~1-2h | ✅ 必须做 |
| 房间系统（创建/加入/开始） | ~3-4h | ✅ 必须做 |
| 角色能力（1 个被动技能） | ~2-3h/每个 | ⚠️ 选做，每个角色加时间 |
| 装备系统 | ~4-6h | ❌ 不推荐 |
| 响应链（杀→闪→桃） | ~6-8h | ❌ 不推荐 |
| 拖拽出牌 | ~3-4h | ❌ 用点击替代 |
| 卡牌动画 | ~3-5h | ❌ 用即时结算替代 |
| AI 对手 | ~6-8h+ | ❌ 绝对不做 |
| 重连机制 | ~8h+ | ❌ 绝对不做 |
| 大厅聊天 | ~2-3h | ❌ 不做，用第三方语音 |
| 排行榜/数据统计 | ~4h+ | ❌ 不做 |
