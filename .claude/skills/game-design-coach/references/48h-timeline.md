# ⏱ 48 小时时间线规划

> 读取此文件场景：需要规划或调整 jam 时间线时。
> Unity + C# + Mirror + 卡牌游戏，4 人团队（2 策划 + 1 程序 + 1 美术）

---

## 一、核心理念

### 48 小时的残酷真相
```
前 8 小时决定成败——如果前 8 小时没有可以跑的核心循环，
后面 40 小时都在追债。
```

### 时间分配黄金比例
```
开发：60%（29h）
打磨：20%（10h）
发布：10%（5h）
休息：10%（4h）← 一定要睡觉！不睡觉的第 40 小时产出是负的
```

### "砍"的哲学
```
如果你觉得时间够 → 砍一半
如果你觉得砍完刚好 → 再砍一半
如果你觉得砍完剩的东西太少了 → 这就是 48h 能做的东西，认命吧
```

## 二、详细时间线

### Phase 0：设计冲刺（第 0-2 小时）

```
目标：把 GDD 定下来，不改了
参加者：全员（策划主导，程序提意见）

0:00 - 0:30 —— 头脑风暴主题（别超过 30 分钟）
0:30 - 1:00 —— 确定核心机制和一句话 Hook
1:00 - 1:30 —— 填充 Game Design Canvas 到 80%
1:30 - 2:00 —— 用 grill-me 压测最有争议的 2 个决策

产出：✅ 完整的 Game Design Canvas
     ✅ 确定主题、核心机制、卡牌类型
     ✅ 决定角色数量和能力
     ❌ 不需要：精确数值、平衡表、所有卡牌的具体效果（留 4-5 张 MVP 牌就够了）

⚠️ 红线：第 2 小时结束时，设计必须锁定。
之后只裁剪不新增。任何新想法写进"未来扩展"清单，不要改当前设计。
```

### Phase 1：MVP 冲刺（第 2-10 小时）

```
目标：能跑起来的核心循环，至少 2 人可联机
核心工作流：程序专注写核心代码，策划写卡牌文案、调数值，美术出 UI 草图和白盒资源

2:00 - 5:00 —— 网络基础 + 房间系统（Mirror RoomManager）
5:00 - 7:00 —— 核心循环（回合系统 + 基本出牌 + 伤害结算）
7:00 - 9:00 —— 基础卡牌（3-5 张 MVP 卡牌）
9:00 - 10:00 —— 联网测试 + Bug 修复

产出：✅ 2 人可以联机
     ✅ 能走完完整回合
     ✅ 3-5 张牌可用
     ✅ 有伤害和胜负判定
     ❌ 不需要：好看的 UI、动画、音效、角色能力

⚠️ 红线：第 10 小时如果还不能联机打一局 → 缩减卡牌到 2 张，
          简化回合逻辑，无论如何先跑起来。
```

### Phase 2：内容填充（第 10-18 小时）

```
目标：增加内容和多样性

10:00 - 12:00 —— 完整的卡牌（8-15 张）+ 卡牌效果逻辑
12:00 - 14:00 —— 角色能力（每个角色 1 个独特效果）
14:00 - 16:00 —— 美术出 UI 资源，程序绑定集成
16:00 - 18:00 —— 集成测试 + Bug 修复

产出：✅ 15 张左右卡牌
     ✅ 3-4 个有独特能力的角色
     ✅ 可读的 UI
     ❌ 不需要：精美 UI、特效、音效、动画

⚠️ 红线：如果 Phase 1 超时了 → 砍角色能力，所有角色裸体上阵
          差异性只通过血量/手牌数体现。
```

### 🛌 休息时间（第 18-20 小时）

```
🔴 必须休息。不休息的话后面 28 小时你会写出不可维护的代码
   和越来越烂的设计决策。

- 至少睡 4 小时（如果你一个人，也要睡）
- 团队轮班可以压缩到 2 小时睡眠但必须有
```

### Phase 3：打磨迭代 1（第 20-28 小时）

```
目标：完善 + 加一点 polish

20:00 - 22:00 —— Bug 修复（Phase 1-2 积压的 bug）
22:00 - 24:00 —— 卡牌平衡调整（数值 tuning）
24:00 - 26:00 —— UI 美化（美术主导调色、布局、卡牌精修）
26:00 - 28:00 —— 测试 + Bug 修复

产出：✅ 所有已知 bug 修复
     ✅ 卡牌数值基本平衡
     ✅ UI 顺眼
     ❌ 不需要：特效、音效、过场动画
```

### Phase 4：打磨迭代 2 / 如果还有时间（第 28-36 小时）

```
这个阶段是"奖励时间"——前面的进度如果良好，可以加东西。

可选内容（按优先级排列）：
1. 🥇 音效基础（出牌音效、受伤音效）
2. 🥇 简单特效（卡牌发光、伤害数字弹出）
3. 🥈 PASS 按钮和回合结束确认
4. 🥈 弃牌阶段 UI
5. 🥉 卡牌小图/图标
6. 🥉 BGM

🔴 注意：如果前面延时了 → 这个阶段全部用来修 bug。
```

### Phase 5：发布冲刺（第 36-44 小时）

```
目标：出一个可发布的 Build

36:00 - 38:00 —— Build 打出来（Windows/Mac 双平台）
38:00 - 40:00 —— 自己测试 Build（至少 3 局完整游戏）
40:00 - 42:00 —— 找朋友测试（远程联机！）
42:00 - 44:00 —— 修发布版的 bug

注意事项：
- !!! 找别人测试，不要自己测。你自己玩习惯了
  什么都觉得是 feature 不是 bug
- 远程联机测试必须在真实的网络条件下测（不是 localhost）
- 记录所有测试中发现的问题，只修 P0
```

### Phase 6：提交（第 44-48 小时）

```
44:00 - 45:00 —— 写 itch.io 页面文案（策划的工作！）
45:00 - 46:00 —— 截图 + 录 GIF（展示核心玩法）
46:00 - 47:00 —— 最终 Build 上传
47:00 - 48:00 —— 提交、写 Game Design Document 摘要、放松

提交清单：
✅ WASM/Windows Build 上传到 itch.io
✅ 截图（至少 3 张）
✅ GIF 或短视频（30 秒展示核心循环）
✅ 游戏描述文字
✅ 操作说明
✅ 团队名单
```

## 三、危险信号 & 应对

### 如果出现以下信号，立刻砍功能

| 时间点 | 危险信号 | 行动 |
|--------|---------|------|
| 第 6 小时 | 房间系统还没跑通 | 砍到最简：硬编码 2 人直连，跳过房间 UI |
| 第 10 小时 | 还不能打完一局 | 砍到最简：2 人、2 种牌（攻击/防御）、无角色能力 |
| 第 14 小时 | 卡牌效果逻辑还有一半没写 | 砍掉效果最复杂的 3 张牌，保证基础牌完整 |
| 第 24 小时 | 还有未修复的致命 bug | 砍掉 Phase 4（打磨迭代 2）全部用来修 bug |
| 第 36 小时 | 还有已知的崩溃 bug | Build 做一个"debug 模式"的版本提交，标注已知问题 |
| 第 42 小时 | 联机测试发现房间系统无法工作 | 准备一个"单机版"作为后备提交方案 |

### "2 小时急救" — 任何时候时间不够了

```
直接砍到最简可玩版本（约 2 小时能改完）：
1. 去掉所有角色能力（所有角色一样）
2. 保留 2 种牌（攻击 + 防御）
3. 保留胜负判定
4. UI 用 Unity UI 默认白盒
5. 不要音效、特效、动画

这个版本虽然简陋，但它是完整的游戏。
完成度 > 复杂度。一个能玩的简单游戏 > 半成品的大作。
```

## 四、团队协作建议

### 策划分工建议（2 个策划）

```
2 个策划的建议分工：

策划 A（系统设计）：
- 主导 Game Design Canvas 填充和维护
- 核心机制设计 + 卡牌系统框架
- 组织 grill-me 决策会议
- 和程序对接技术可行性

策划 B（内容 + 测试）：
- 卡牌文案、命名、故事背景
- 数值表维护和平衡调整
- 测试清单编写 + 主导联机测试
- itch.io 页面、截图、GIF、发布文案

并行工作模式：
- Phase 0：两人一起完成设计冲刺，快速锁定 GDD
- Phase 1-2：A 跟程序对接系统设计，B 产出全部卡牌内容和测试用例
- Phase 3-6：A 做平衡调整和数值 tuning，B 主导测试和发布准备

⚠️ 注意：开发阶段策划不要闲着等程序写完再反馈。
          程序写代码的时候，策划 B 应该在生产内容，策划 A 在做明日规划。
          2 个策划的优势就是设计和内容可以完全并行。
```

### 程序员分工建议（1 个程序 — 严格执行优先级）

```

只有一个程序员，所有任务必须按优先级串行执行。不要并行切换分支。

Priority 1（Phase 1 MVP，必须完成）：
- Mirror 房间系统 + 回合管理
- 最基本的出牌 + 伤害结算逻辑
- 2 人联机能跑完一局

Priority 2（Phase 2 内容填充）：
- 卡牌效果引擎（泛化，支持加新牌）
- 角色能力逻辑
- 美术资源接入 + UI 绑定

Priority 3（Phase 3+ 打磨）：
- 动画/特效
- 音效集成
- UI 细节

🔴 红线：Phase 1 期间不要碰 UI 代码——用 Unity 默认 UI 文字代替。
    策划和美术做的 UI 素材先存着，Phase 2 再集成。
    网络 + 核心循环跑通之前，一切都不重要。
```

### 美术分工建议（1 个美术）

```
核心原则：美术资源按"够用就行"的标准出，不要追求精修。48h 玩家不关心像素是否完美。

各阶段产出：

Phase 1（MVP，第 2-10h）：
- UI 布局草图（Figma 或纸笔 — 给策划 A 确认布局，给程序员做参考）
- 卡牌白盒占位（纯色方块 + 文字说明即可，不要花时间画图）
- 配色方案初稿（2-3 套，选一套快速定下来）

Phase 2（内容填充，第 10-18h）：
- 卡牌图示（简笔画/扁平风格，每张 ≤ 30 分钟）
- 角色立绘（每个角色 ≤ 1 小时）
- UI 资源（血量图标、手牌区底框、按钮样式）
- 背景/桌布纹理（简单的渐变或几何图案）

Phase 3+（打磨，第 20h+）：
- UI 美化（配色统一、字体、按钮 hover 效果）
- 卡牌边框/稀有度区分
- 特效素材（Unity Particle System 用默认资源微调）

⚠️ 不要做的事：
- 逐帧动画（48h 做不完）
- 高清插图（分辨率不是 jam 的重点）
- 3D 模型（2D 卡牌够用，不要额外工作量）
- Icon 精修（用 emoji 或简单几何代替即可）

🔴 红线：Phase 1 结束前不要出"成品图"——先出白盒占位。
    程序核心循环跑通后，美术再替换正式资源。
```

### 沟通规则
```
- 每 2 小时同步一次进度（3 分钟站立会）
- 任何超过 1 小时的技术难题 → 降级方案（最简单的方案先上了再说）
- 策划说"这个效果做不了" → 程序说"可以做，但要 X 小时" → 大家决定做不做
- 吵架了 → 策划拍板设计问题，主程拍板技术问题
